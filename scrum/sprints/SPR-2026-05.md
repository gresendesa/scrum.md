## PO Questions

```yaml
section: sprint.SPR-2026-05.po-questions
title: PO Questions
status: active
owner: Guilherme
source_backlog_item: B-006
questions:
  - Should the first signing policy use a cryptographic signature, a checksum-only declaration, or a placeholder policy for local development?
  - If cryptographic signing is required, what key format, key location, and signing command should `smd pack` support first?
  - Should unsigned packages be rejected, warned, or allowed only with an explicit flag?
  - Which manifest fields are mandatory for the first version, such as name, version, description, author, template engine, files, instructions, compatibility, and entrypoints?
  - Should `manifest.yaml` replace the current `template.yml` name or should both names be supported during migration?
  - Should the signature cover only file contents or also zip metadata such as file order and timestamps?
  - Should `smd template render` validate the manifest and signature immediately, or should verification be a separate future command?
tags: [planning, questions, po, manifest, signature]
```

Implementation must not start until the PO answers these questions.

## Risk

```yaml
section: sprint.SPR-2026-05.risk
title: Risk
status: active
owner: Guilherme
classification: medium
reason: Package manifest and signing policy define a public package contract that future `smd init` behavior will depend on.
mitigations:
  - Require PO answers before implementation starts.
  - Keep the first signing policy narrow and explicit.
  - Keep web download, remote trust registry, and key server integration out of scope.
tags: [risk]
```

## Guardrails

```yaml
section: sprint.SPR-2026-05.guardrails
title: Guardrails
status: active
owner: Guilherme
tags: [guardrails]
```

* Do not implement `smd init` in this sprint.
* Do not implement web download, upload, remote trust registry, cache, or key server behavior.
* Do not render templates during `smd pack`.
* Do not implement cryptographic key signing.
* Do not validate package signatures in `smd template render` during this sprint.
* Keep output deterministic and JSON-friendly.
