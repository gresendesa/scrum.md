## Objective

```yaml
section: backlog.item.B-006.objective
title: Objective
status: active
owner: Guilherme
tags: [objective]
```

Create an `smd pack` operation that receives a local template package directory and produces a `.zip` containing that directory's contents, package declaration manifest, and checksum signature metadata according to the PO-approved policy.

## Scope

```yaml
section: backlog.item.B-006.scope
title: Scope
status: active
owner: Guilherme
in_scope:
  - Add a public `smd pack <directory>` command.
  - Accept an output path option for the generated `.zip`.
  - Define the required `manifest.yaml` pattern that declares the package.
  - Define the initial zip signing policy as checksum-only.
  - Include the input directory contents using deterministic relative paths.
  - Validate that the package directory contains a valid `manifest.yaml` before packing.
  - Generate `SIGNATURE.yaml` with checksums covering file contents, relative paths, and deterministic file order.
  - Reject packages that cannot produce signature metadata.
  - Reject missing input directories and unsafe output paths with structured errors.
  - Preserve the stable JSON envelope for success and failure paths.
  - Report the generated zip path, manifest identity, signature status, and packed file list in JSON output.
  - Add automated CLI tests for successful pack and invalid input.
out_of_scope:
  - Web download or upload of template packages.
  - Remote trust registry, key server integration, cache, or remote version policy.
  - Full cryptographic supply-chain framework beyond the PO-approved first signing policy.
  - Cryptographic key management or private-key signing.
  - Rendering templates during the pack operation.
  - Validating package signature during `smd template render`.
tags: [scope, cli, packaging, zip, manifest, signature]
```

## Planning Questions

```yaml
section: backlog.item.B-006.planning-questions
title: Planning Questions
status: active
owner: Guilherme
questions:
  - Should the first signing policy use a cryptographic signature, a checksum-only declaration, or a placeholder policy for local development?
  - If cryptographic signing is required, what key format, key location, and signing command should `smd pack` support first?
  - Should unsigned packages be rejected, warned, or allowed only with an explicit flag?
  - Which manifest fields are mandatory for the first version, such as name, version, description, author, template engine, files, instructions, compatibility, and entrypoints?
  - Should `manifest.yaml` replace the current `template.yml` name or should both names be supported during migration?
  - Should the signature cover only file contents or also zip metadata such as file order and timestamps?
  - Should `smd template render` validate the manifest and signature immediately, or should verification be a separate future command?
tags: [planning, questions, po, manifest, signature]
```

During sprint planning, the PO must answer these questions before implementation starts so the command does not invent package trust semantics.

## Acceptance Criteria

```yaml
section: backlog.item.B-006.acceptance-criteria
title: Acceptance Criteria
status: active
owner: Guilherme
criteria:
  - `smd pack <directory> --output <file.zip> --json` creates a zip file from a package directory.
  - The package directory must contain a valid `manifest.yaml` following the PO-approved pattern.
  - `manifest.yaml` is the only accepted manifest filename for `smd pack`.
  - The zip contains the input directory contents with safe relative paths and deterministic ordering.
  - The command generates `SIGNATURE.yaml` using checksum-only metadata.
  - The checksum metadata covers file contents, relative paths, and deterministic file order.
  - The command rejects packages that cannot produce checksum signature metadata.
  - The command returns the stable JSON envelope with the output zip path, manifest identity, signature status, and packed files.
  - Missing or non-directory input returns a non-zero exit code and structured error.
  - Missing or invalid `manifest.yaml` returns a non-zero exit code and structured error.
  - Existing output files are not overwritten unless an explicit force option is provided.
  - Automated tests cover success, invalid input, invalid manifest, checksum signature behavior, and overwrite protection.
tags: [acceptance, cli, tests, manifest, signature]
```
